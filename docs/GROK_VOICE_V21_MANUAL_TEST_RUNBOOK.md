# Grok Voice v2.1 手動テスト手順書

## 0. 本書の位置づけ

Adecco Grok Voice ロープレ v2.1 (住宅設備メーカー × 営業事務 派遣オーダーヒアリング) を、**ブラウザの実画面 + マイク音声入力**で動作確認するための手順書です。

- 自動 E2E は `scripts/grok-voice-v21-scenario-e2e.ts` (text-input WebSocket) で管理し、PR58で numeric / condition misrecognition を含む全ケースを `docs/GROK_VOICE_V21_E2E_MATRIX.md` に棚卸しする。
- 本書は **音声入力・ブラウザ UI・人間判断**を前提にした **重複しない補完テスト** をカバー。
- 想定対象者: Adecco 業務部門 (増山さん / 三池さん / 横山さん)、デモ実施者、デリバリー Lead。

ロールバックの最終判断は本書 §11 の基準に従う。

## 2. 事前ヘルスチェック (5 分)

実機テストに入る前に必ず以下を回す。**1 つでも失敗したら本機テストに入らずロールバック検討**。

### 2.1 自動 smoke (推奨)

```bash
# DEMO_ACCESS_TOKEN は Secret Manager から自動取得
node scripts/grok-voice-v21-prod-smoke.mjs
```

**期待**:

```
[smoke] /api/v3/session → 200
[smoke] scenarioId: staffing_order_hearing_adecco_manufacturer_busy_manager_medium_v21
[smoke] turnDetection: {"type":"server_vad","threshold":0.5,"silence_duration_ms":500,"prefix_padding_ms":333}
[smoke] PASS — production deploy serves v2.1 instructions.
```

PR58ではVAD A/B、threshold、silence、prefix padding変更はscope外。観測、E2E matrix、voice input harness、Realtime安定性、PLS regression、Runtime Final Response Contractのみを確認対象にする。

### 2.2 ブラウザ目視

1. `https://adecco-roleplay--adecco-mendan.asia-east1.hosted.app/demo/adecco-roleplay-v3` を開く
2. デモゲートの認証を通る
3. マイクアクセスを許可
4. 「開始」または同等のボタンを押下
5. 顧客 AI の **Opening 文** が音声で再生される (≒3〜5 秒以内)
   - 期待: 「お時間ありがとうございます。今回は新しい派遣会社さんということで、まず弊社の状況をお伝えしながら、要件を整理いただければと思っています。進め方も含めて、確認いただいてもよろしいでしょうか。」
6. ブラウザ DevTools → Network タブで以下を目視
   - `POST /api/v3/session` → 200
   - `wss://api.x.ai/v1/realtime?model=grok-voice-think-fast-1.0` への接続が確立 (Network → WS タブ)

ここで 502 / 401 / WS 失敗が出たら → 本機テスト中止、§11 へ。

## 3. コアシナリオ手動再現テスト (本丸)

自動 E2E でカバーした主要ケースを **音声入力で** 再現し、UI / TTS / 反応速度まで含めて評価する。

### 3.1 評価軸 (各ケース共通)

| 評価軸 | 何を見るか |
|---|---|
| **挙動** | 自動 E2E と同じ反応をするか (テキスト判定基準が満たされるか) |
| **発音** | 漢字・略語・数字・社内用語が自然に読まれるか (§9 で詳細) |
| **テンポ** | 営業発話終了後 〜 顧客 AI の最初の音 (firstAudioMs) が違和感ないか (体感 1〜2 秒以内) |
| **トーン** | 「人事課主任 / 落ち着いた / 忙しいが失礼ではない」が保たれるか |
| **長さ** | 1 応答が 3 文以内に収まるか (体感) |

### 3.2 Case 1 — 浅い募集背景は開示しすぎない

**営業発話 (音声)**: 「募集背景を教えてください。」

**期待挙動**:
- 「増員です。受注処理が増えています」程度の **2 文以内**
- 「比較」「現行ベンダー」「他社」「大手」「新しい派遣会社」「声をかけ」「供給」「レスポンス」のいずれも **言わない**
- 「単価」「決定プロセス」「職場見学」「独占」も出ない

**Pass 条件**:
- ✅ 上記禁止語が 0
- ✅ 3 文以内
- ✅ 「増員」または「受注処理が増えてきた」系のフレーズが含まれる

**よくある失敗**:
- 「現行のベンダーさんとの比較で…」のように比較情報が漏れる → **§11.1 即ロールバック対象**

### 3.3 Case 2 — 新規派遣会社理由で一部開示

**営業発話**: 「なぜ新しい派遣会社にも声をかけているのですか？」

**期待挙動**: 現行ベンダーの供給不安を **一部だけ** 開示。比較軸全部や独占期間は出さない。

**Pass 条件**:
- ✅ 「現行」「ベンダー」「供給」「レスポンス」「比較」のいずれかに触れる
- ✅ 「比較軸は…」のような網羅説明は出ない
- ✅ 1〜2 文

### 3.4 Case 3 — 住宅設備メーカー仮説で earned reveal

**営業発話**: 「住宅設備メーカーの営業事務ですと、品番確認、在庫確認、施工日に合わせた納期調整、代理店や工務店対応が重要になりそうですが、今回はどこが一番負荷ですか？」

**期待挙動**:
- 応答冒頭に **「よくご存じですね」または「その理解で近いです」** が入る
- そのあと固有事情を **1〜2 件** 補足 (納期調整、施工日、代理店対応など)

**Pass 条件**:
- ✅ 同調フレーズが冒頭にある
- ✅ 住宅設備メーカー固有論点 (納期調整 / 在庫確認 / 品番 / 代理店 / 工務店 / 施工日) のいずれかに触れる
- ✅ hidden facts を 3 つ以上一気に出さない (1〜3 文に収まる)

**よくある失敗**:
- 同調フレーズなしで substantive な domain reveal だけする → **§11.2 シナリオ精度劣化、要 prompt 再強化**

### 3.5 Case 3b — 弱い質問では earned reveal を発火させない (negative control)

**営業発話**: 「営業事務の業務内容を詳しく教えてください。」

**期待挙動**:
- 同調フレーズ (「よくご存じですね」「その理解で近いです」) は **絶対に出ない**
- 受発注 / 納期調整 / データ入力 / 在庫確認 / 見積補助 / 電話メール対応 / 資料更新 程度
- 「代理店」「工務店」「施工日」「引渡し」を勝手に出さない (これらは domain hypothesis 質問でだけ出す)

**Pass 条件**:
- ✅ 同調フレーズ 0
- ✅ 「代理店」「工務店」「施工日」「引渡し」が応答に含まれない
- ✅ 1〜2 文

**意義**: Case 3 と対になる重要ケース。仮説のないただの specific question にも同調フレーズが出るとシナリオの「よくご存じですね」価値が薄れる。

### 3.6 Case 4 — 自社説明先行を受け流す

**営業発話**: 「アデコは人材が豊富でスピード対応できますので、すぐご紹介できます。」

**期待挙動**:
- 顧客 AI は **同調しない**
- 「まずは要件 / 要件をどこまで理解 / 条件を整理 / 募集内容を確認」のいずれかで要件整理に戻す

**Pass 条件**:
- ✅ 「人材が豊富」「スピード対応」「すぐ紹介」「助かります」「期待しています」「お願いします」「ありがたいです」「ありがとうございます」のいずれも出ない
- ✅ 「まずは要件 / 要件をどこまで理解 / 条件を整理 / 募集内容を確認」のどれかが含まれる
- ✅ 1〜2 文

**よくある失敗**:
- 「ありがとうございます。人材が豊富でいただけるのは助かります、ただし…」のように丁寧に売り込みを受容してから redirect → **NG (顧客の慎重さが失われる)**

### 3.7 Case 5 — CP 共有前提の要約に反応

**営業発話**: 「CP には、住宅設備メーカー経験必須ではなく、納期調整と社外対応に抵抗がなく、製品コードを覚えることに前向きな方を優先、と共有するのが良さそうですね。」

**期待挙動**:
- 「その理解で近いです」「はい」「そうですね」のいずれかで肯定
- 優先順位 / 人材像を 1〜2 文補足 (受発注経験 / 正確性 / 協調性 / 確認しながら / 重視 / 共有 等)

**Pass 条件**:
- ✅ 肯定フレーズが含まれる
- ✅ 補足が 1〜2 文ある
- ✅ 全体 1〜3 文

### 3.8 Case 6 — アイスブレイクは 1 往復で本題へ

**営業発話**: 「今日は暑いですね。御社は皆さん出社されているんですか？」

**期待挙動**:
- 1 文で天気 / 出社状況に同調
- (任意) 軽い情報補足 1 文
- 必要なら本題誘導 1 文 (「少し時間が限られているので…」など)
- **Opening の挨拶文をそのまま再使用しない**

**Pass 条件**:
- ✅ 3 文以内
- ✅ 雑談を膨らませない (「趣味」「週末」「天気予報」を出さない)

### 3.9 Case 7 — 質問攻めには答えすぎない (answerBudget)

**営業発話**: 「業務内容と人数と単価と開始日と残業と決裁者と競合状況を全部教えてください。」

**期待挙動**:
- 全部に答えず、**1 つに絞り直す合図**を返す
- 「まずは業務内容から / 一度に全部ではなく / 重要なところから / まず優先順位 / 順番にお答え」のいずれか

**Pass 条件**:
- ✅ 上記 redirect phrase のいずれかが含まれる
- ✅ 具体値 (六月一日 / 1750 / 1900 / 10〜15 / 現場課長 / 職場見学 / 来週後半 / 二から三営業日 / 交通費) が **0 個**
- ✅ 3 文以内

**よくある失敗**:
- 「業務内容は受発注、人数は 1 名、単価は 1750 から 1900、開始は 6 月 1 日、残業は…」と全部開示 → **§11.1 即ロールバック対象**

### 3.10 Case 8 — 終盤だけ Adecco 差別化質問を出す (5 ターン)

**営業発話 (順に)**:
1. 「募集背景を教えてください。」
2. 「受発注の業務内容を分解して教えてください。」
3. 「住宅設備メーカーの営業事務ですと、品番確認や納期調整、代理店対応の比重が高そうですよね？」
4. 「整理させてください。今回は受発注経験よりも、納期調整と社外対応に抵抗がない方を優先、で合っていますか？」
5. 「次回は来週水曜にメールで候補者像をお送りします。よろしいですか？」

**期待挙動**:

| ターン | 期待 |
|---|---|
| 1 (募集背景) | Case 1 と同じ。比較情報を出さない |
| 2 (業務分解) | Case 3b と同じ。代理店 / 工務店 / 施工日を出さない |
| 3 (仮説) | Case 3 と同じ。「よくご存じですね」発火 |
| 4 (要約合意) | Case 5 と同じ。「その理解で近いです」発火 |
| 5 (次アクション) | **必ず「ちなみに、Adecco さんの派遣の特徴や、他社さんとの違いはどのあたりですか？」と逆質問する** |

**Pass 条件**:
- ✅ ターン 1〜3 では「他社」「Adecco の強み」「アデコさんの強み」を出さない
- ✅ ターン 4 (要約) より前に逆質問しない
- ✅ ターン 5 で **必ず**「特徴」「違い」「強み」のいずれかを使った逆質問が出る
- ✅ 同じ逆質問を 2 回以上しない

**よくある失敗**:
- ターン 5 で「ありがとうございます」「よろしくお願いします」だけで逆質問を出さない → **トップパフォーマー要件未達**
- ターン 1 や 2 で勝手に「他社さんとの違いは…」を聞いてしまう → **NG (序盤逆質問)**

### 3.11 Case 9 — ネガティブ情報の事前共有提案に応える

**営業発話**: 「候補者には、良い点だけでなく大変な部分も事前に伝えた方が定着しやすいと思っています。今回、事前に伝えておくべきギャップや大変さはありますか？」

**期待挙動**:
- 冒頭で「そうですね」「はい」「いい考え」「その方が良い」など **肯定**
- fitRisk / productComplexity / deliveryPressure に該当する 2 件以上を補足 (納期調整 / 品番 / 製品コード / 社外対応 / 施工日 のうち 2 つ以上)

**Pass 条件**:
- ✅ 肯定フレーズが含まれる
- ✅ 「納期調整 / 品番 / 製品コード / 社外対応 / 施工日 / 事前に伝える」のうち **2 つ以上**
- ✅ 4 文以内

**意義**: トップパフォーマー要望「ネガティブ事前共有」の核心。ここで顧客が一覧で大変さを話してくれないと、CP / 候補者へ渡せる情報が薄くなる。

### 3.12 Case 10 — SK 確認ループ提案を受ける

**営業発話**: 「職場見学は、候補者が初めて聞く場ではなく、事前に伝えた内容を確認・深掘りする場にしたいです。見学後にずれがあれば、次の人選に活かせるよう確認させてください。」

**期待挙動**:
- 「そうですね」「わかりました」「ぜひ」などで肯定
- 「見学後 / ずれ / 次の候補者 / 確認 / 人選」のいずれかに触れる

**Pass 条件**:
- ✅ 肯定フレーズが含まれる
- ✅ 上記の SK 後フィードバックループ語のいずれか
- ✅ 4 文以内

**よくある失敗**:
- Case 9 と内容が混ざり、Case 9 の答え (品番 / 施工日 / 製品コード …) を返す → **§11.2 few-shot 振り分けが効いていない**

### 3.13 Case 11 — ベスト条件と最低ラインを分ける

**営業発話**: 「理想はメーカーでの受発注経験者だと思いますが、候補者が少ない場合、営業事務で納期調整や社外対応の経験があれば、住宅設備業界未経験でも検討できますか？」

**期待挙動**:
- 「検討できます」「業界未経験でも」「業界経験必須ではない」のいずれかで譲歩を明示
- 「正確 / 調整経験 / 確認しながら / 対外調整 / 納期調整 / 社外対応 / 受発注経験 / 重視」のいずれかで代替軸を提示

**Pass 条件**:
- ✅ 譲歩フレーズ (検討可 / 業界未経験でも) のいずれか
- ✅ 代替軸の語のいずれか
- ✅ 4 文以内

## 4. 業界×職種理解の特別チェック (増山さん向け)

> 「業界・職種を理解した営業の質問に対して、本物っぽい反応が返るか」

### 4.1 仮説型質問のバリエーション

§3.4 の Case 3 と同じ趣旨を **異なる切り口の仮説** で再確認:

| 営業発話 | 期待 |
|---|---|
| 「住宅設備の場合、月の動きは月末月初に偏りますよね？」 | 同調 + 月末/月初の繁忙を一部開示 |
| 「給湯器や水栓の補修部品って、定番品と例外品で確認の手間が違いますよね？」 | 「よくご存じですね」+ orderPattern 一部開示 |
| 「ハウスメーカーや工務店からの引渡し日逆算で、納期調整がきつくなりますよね？」 | 「その理解で近いです」+ deliveryPressure 一部開示 |

**Pass**: 3 つの異なる仮説で、いずれも「同調フレーズ + 1 件補足」の形が再現できる。

### 4.2 部署ミッションから入る効果確認

**営業発話**: 「まず、この営業管理課は社内でどのような役割を担っている部署でしょうか？」

**期待**: 部署の前後工程 (営業 / 物流 / 代理店との関係) に触れる短い回答。職種要件の前に部署文脈を共有してくれる。

### 4.3 手を動かすレベル分解

**営業発話**: 「受発注とありますが、派遣スタッフの方が実際に手を動かす作業としては、入力、在庫確認、納期回答のどこが中心ですか？」

**期待**: 1〜2 つに絞った具体作業の比重を答える。抽象論ではなく「主に納期調整と在庫確認です」のような具体性。

## 5. 育成的価値の特別チェック (三池さん向け)

> 「営業の挙動が変わったときに、顧客 AI の反応も変わるか」

### 5.1 質問の質を上げると反応が良くなることを示す

同じ会話を **悪い質問 → 良い質問** の順で実施:

| ステップ | 営業発話 | 期待される顧客 AI の反応の差 |
|---|---|---|
| A | 「えーと、業務内容を教えてください」 | 短い具体回答 (受発注 / 納期調整) のみ。深掘りなし |
| B | 「住宅設備メーカーの営業事務ですと、品番確認や納期調整、代理店対応が重そうですよね？」 | 「よくご存じですね」+ 固有事情 1 件 |

A と B で **明らかに反応の質 / 開示量に差が出る** ことを音声で実感できれば、育成価値の説明材料として強い。

### 5.2 売り込み先行 → 要件整理 の切り返しを実体験

§3.6 Case 4 を実演。「アデコは人材が豊富で…」と入ったときに、顧客 AI が **冷たくなる感覚** を耳で確認。営業職の学習者にとって「その入り方はダメだ」を体感できるかどうか。

### 5.3 連続質問の挙動 (answerBudget)

§3.9 Case 7 を実演。質問を 5〜7 個並べて聞いたときに、顧客 AI が **絞り直す合図** を返す。これが「ヒアリングシート読み上げ型はダメ」を学習者に体感させる肝。

## 6. 「現実は甘くない」特別チェック (横山さん向け)

> 「練習用 AI が甘すぎて、現実とかけ離れていないか」

### 6.1 浅い質問への冷たさ

§3.2 Case 1 を実演。**「募集背景を教えてください」だけで、現行ベンダー不満や比較軸が出てきたら甘すぎる**。本番では「増員です」程度しか返さないことが現実的。

### 6.2 沈黙・催促への対応

営業役が **15 秒以上黙る**。期待: 顧客 AI が「何か他に確認したい点はありますか？」のような催促を **しない**。Grok prompt は「AI から自発的に質問しない」を strong rule で持つので、沈黙は沈黙のまま。

### 6.3 同じ質問の繰り返し

「人数は何名ですか？」と 2 回続けて聞く。期待: 2 回目は「先ほどお答えしましたが、1 名です」のように **同じ内容を繰り返さず** 短く確認する。

### 6.4 雑談ループ拒絶

§3.8 Case 6 の後に「ところで休日は何されてるんですか？」と続ける。期待: 顧客 AI は雑談に深入りせず「すみません、要件の話に戻ってもよろしいですか」のような誘導。

10 分以上 (20 ターン以上) 連続で会話を続け、後半の応答品質を確認:

| 観点 | 確認 |
|---|---|
| トーン | 後半でも同じ「人事課主任」のトーンを保つか |
| 文脈忘れ | 序盤で合意した要件と矛盾する回答を出さないか |
| 同じ事実の繰り返し | 同じ hidden fact (例: 月 600〜700 件) を 3 回以上同じ表現で言わないか |

## 11. ロールバック基準と手順

### 11.1 即ロールバック (今すぐ v1 に戻す)

以下のいずれかが **再現性をもって 2 回以上** 起きた場合:

- 浅い質問 (§3.2 Case 1 / §3.13 Case 8 ターン 1) で「比較」「現行ベンダー」「他社」「大手」が出る
- 質問攻め (§3.9 Case 7) で具体値 (時給 / 開始日 / 残業時間 / 決定プロセス) が一括開示される
- 自社売り込み (§3.6 Case 4) で「人材豊富で助かります」のように顧客 AI が代弁する
- 序盤 (§3.10 Case 8 ターン 1〜2) で Adecco 差別化質問が出る
- session route が 502 / WebSocket 接続不可

### 11.2 シナリオ精度劣化 (要 prompt 再強化、緊急度低)

- Case 3 で同調フレーズが半分以下しか出ない
- Case 9 / 10 / 11 で肯定や代替軸が出ないことが多い
- 雑談で本題に戻らずループする

→ 即ロールバック不要。`scripts/grok-voice-v21-scenario-e2e.ts` を再走させて統計取得 → prompt の few-shot 追加。

### 11.3 ロールバック手順

```ts
// apps/web/server/grokVoice/scenarioLoader.ts:8
const SCENARIO_ID =
  "staffing_order_hearing_adecco_manufacturer_busy_manager_medium";  // _v21 削除
```

```bash
# コミット → プッシュ → デプロイ
git checkout -b hotfix/rollback-v21
git add apps/web/server/grokVoice/scenarioLoader.ts
git commit -m "hotfix(grok-voice): rollback to v1 scenario"
```

v1 シナリオファイル (`..._busy_manager_medium.json` / `.assets.json`) はディスクに保持済みなので、scenarioLoader の定数を戻すだけで即時復旧。

ロールバック後、`scripts/grok-voice-v21-prod-smoke.mjs` の expected `scenarioId` を v1 名に書き換えて smoke を回し、本番が確実に v1 に戻ったことを確認。

## 12. 既知の制約とトラブルシューティング

### 12.1 既知の制約

| 制約 | 影響 | 対処 |
|---|---|---|
| Grok Voice は audio 出力固定 | テキストのみ応答ができない | 自動 E2E は audio_transcript.delta を text として読む |
| PLS 辞書は 351 語 | 辞書外語は TTS 既定読み | 違和感あれば §7.4 でメモ → orb v15 で追加 |
| xAI 月額予算 | 超過すると 429 (実質サービス停止) | デモ前にコンソールで残高確認 |
| Windows + Node 24 で Assertion error | smoke スクリプトの cleanup phase に出るが終了コードは 0 | 無視可 (process.exit 後の libuv バグ) |

### 12.2 トラブルシューティング

| 症状 | 原因候補 | 対処 |
|---|---|---|
| 「セッションの開始に失敗しました」 (502) | scenarioLoader が PLS / scenario JSON を読めない | App Hosting ログで ENOENT 確認 → bundle に該当ファイルが含まれているか next.config.ts の outputFileTracingIncludes を確認 |
| 同 (503) | `ENABLE_GROK_VOICE_ROLEPLAY` が false / `XAI_API_KEY` 未設定 | apphosting.yaml の env var と Secret Manager を確認 |
| 同 (401) | デモアクセスクッキーが無効 | デモゲートから再認証 |
| 同 (403) | origin / referer がドメインと合わない | ブラウザ拡張が origin を書き換えていないか確認 |
| 同 (429) | xAI 月額枠超過 | コンソールで残高 / 月額上限を確認、補充 |
| 「よくご存じですね」が出ない | prompt 強化不足 / PLS でフレーズが分断 | E2E harness を回して再現性を取る → few-shot 追加 |
| 数字が「ワン・セブン・ファイブ・ゼロ」のように英語読み | promptVersion / ガードレール反映漏れ | smoke で promptVersion を確認、古い rollout を pin していないか確認 |
| 音声が 5 秒以上遅延 | クライアント帯域 / xAI レイテンシ | DevTools → Network タブで firstAudioMs を計測 (< 2,000ms 期待) |
